통합 게임 데이터 및 하드웨어 최적화 플랫폼 요구사항명세서 (SRS)

1. 소개 (Introduction)

1.1. 제품 범위 및 가치 제안

본 플랫폼은 스팀(Steam) 및 라이엇 게임즈(Riot Games) 등 다수의 유통 플랫폼으로 파편화된 사용자의 게임 데이터(누적 플레이 시간, 업적 등)를 단일 대시보드에 통합하고, 사용자의 하드웨어 제원에 최적화된 그래픽 설정값을 제공하는 것을 목적으로 합니다. PC 플랫폼의 특성상 발생하는 하드웨어 파편화 문제와 그에 따른 최적화 설정 탐색 비용을 최소화하며, 시스템 병목(Bottleneck) 현상을 정밀하게 진단하여 최적의 게임 환경을 구축하는 기술적 가치를 제공합니다.

1.2. 대상 독자

* 개발자 및 시스템 설계자: 시스템 아키텍처, 데이터 처리 로직 및 RESTful API 규격을 참조하여 DB 스키마 설계 및 엔드포인트 구현의 기준으로 삼습니다.
* QA 및 테스터: 본 문서의 수락 기준(Acceptance Criteria)을 바탕으로 기능의 정상 동작 여부와 비기능적 품질 속성을 검증합니다.
* 비즈니스 분석가(BA): 비즈니스 모델이 기술적 요구사항으로 정확히 변환되었는지 확인하고 프로젝트의 범위를 관리합니다.

1.3. 시스템 정의

본 시스템은 외부 게임사의 원시 데이터를 수집·가공하여 '단일 진실 공급원(SSoT)' 역할을 수행하는 **'데이터 중개 플랫폼'**이자, 하드웨어 벤치마크 데이터를 기반으로 최적의 설정값을 매칭하는 **'커뮤니티 기반 최적화 허브'**로 정의됩니다.

2. 시스템 및 기능 요구사항 (System & Functional Requirements)

2.1. 요구사항 기술 규칙

모든 기능 요구사항은 EARS(Easy Approach to Requirement Syntax) 구문을 엄격히 준수합니다.

* 형식: WHEN [이벤트 발생], the [시스템] shall [동작].

2.2. 핵심 기능 상세 명세

2.2.1. 외부 계정 연동 및 데이터 이관

* 요구사항: WHEN 사용자가 OAuth 2.0 인증을 통해 외부 플랫폼(Steam, Riot Games) 계정을 연동할 때, the 시스템은 외부 Web API를 호출하여 사용자의 게임 목록 및 업적 데이터를 수집하고 이를 관계형 데이터베이스(RDBMS)에 저장해야 한다.
* [입력]: OAuth Access Token, User Platform ID, API Key.
* [처리 로직]:
  1. 외부 API(Steam/Riot) 엔드포인트 비동기 호출.
  2. 수신된 JSON 데이터에서 App_ID, Playtime_Forever, Achievement_Status 필드 추출.
  3. 기존 DB 내 사용자 식별자와 매핑하여 Update 또는 Insert 수행.
* [출력]: HTTP 200 OK, Sync_Status: Success, Last_Sync_Timestamp, 연동된 게임 리스트 객체(JSON).

2.2.2. 하드웨어 프로필 정규화 및 등록

* 요구사항: WHEN 사용자가 하드웨어 제원(CPU, GPU, 해상도)을 직접 입력하거나 수정할 때, the 시스템은 입력된 텍스트를 표준 하드웨어 DB 명칭과 정규화(Normalization)하여 사용자 프로필에 저장해야 한다.
* [입력]: CPU 모델명(String), GPU 모델명(String), 해상도(Int x Int), 주사율(Int).
* [처리 로직]:
  1. 사용자 입력 문자열에 대한 정규표현식 검사.
  2. 하드웨어 마스터 테이블과의 유사도 검색을 통한 표준 명칭 매칭.
  3. User_Hardware_Profile 테이블에 정규화된 키값 저장.
* [출력]: Profile_Update_Result: Success, 정규화된 하드웨어 모델명 ID.

2.2.3. 가중치 기반 알고리즘 최적화 매칭

* 요구사항: WHEN 사용자가 특정 게임의 최적화 설정을 요청할 때, the 시스템은 DB 내 프로파일 중 사용자의 하드웨어 환경과 90% 이상 일치하는 환경에서 작성된 설정값을 우선순위로 정렬하여 전송해야 한다.
* [입력]: Target_Game_ID, User_Hardware_Spec_Object.
* [처리 로직]:
  1. 다음 가중치에 따른 유사도 점수 계산: [CPU 세대 및 코어 수(20%)], [GPU 등급 및 VRAM(50%)], [해상도 및 주사율(30%)].
  2. 총합 점수 0.9 이상의 레코드 필터링.
  3. Average_FPS 기준 내림차순 정렬.
* [출력]: Ranked_Settings_List(JSON), 각 설정별 상세 그래픽 옵션값 및 예상 FPS.

2.2.4. 데이터 시각화 엔진

* 요구사항: WHEN 통합 대시보드 페이지가 로딩될 때, the 시스템은 수집된 누적 플레이 시간 데이터를 활용하여 기간별 시계열 차트 및 업적 달성률 현황을 렌더링해야 한다.
* [입력]: User_Play_Logs, Achievement_Stats.
* [처리 로직]:
  1. Recharts 라이브러리 인터페이스에 맞춘 데이터 배열 가공.
  2. 일간/주간/월간 단위 데이터 집계(Aggregation).
* [출력]: 동적 시각화 대시보드(Line Chart, Pie Chart).

2.3. 관리 기능

* 시스템 모니터링 및 데이터 관리: WHEN 관리자가 시스템 관리 콘솔에 접속할 때, the 시스템은 전체 서비스의 트래픽 통계, 하드웨어 프로파일 등록 현황 및 API 통신 에러 로그를 조회할 수 있는 인터페이스를 출력해야 한다.
* 사용자 권한 통제: WHEN 관리자가 특정 사용자의 상태를 변경할 때, the 시스템은 해당 사용자의 Access_Level 필드를 업데이트하고 접근 권한을 즉각적으로 제한해야 한다.

3. 외부 인터페이스 요구사항 (External Interface Requirements)

3.1. 사용자 인터페이스 (UI)

* UX 연속성: React.js 기반의 SPA(Single Page Application) 구조를 채택하여 페이지 전환 시 상태 유실을 방지한다.
* 반응형 레이아웃: Tailwind CSS를 활용하여 모바일, 태블릿, 데스크톱 등 다양한 뷰포트 크기에 대응하는 유연한 그리드 시스템을 적용한다.

3.2. 소프트웨어 인터페이스

* Backend Interface: Node.js(Express.js) 프레임워크를 사용하며, JSON 포맷을 통한 RESTful API 통신을 수행한다.
* External API Specs:
  * Steam Web API: IPlayerService/GetOwnedGames 등의 인터페이스를 통한 데이터 수신.
  * Riot Games API: 전적 및 사용자 활동 데이터 수집을 위한 엔드포인트 연동.

3.3. 하드웨어 인터페이스

* 데이터 정규화 인터페이스: 웹 환경의 보안 제약으로 인해 하드웨어를 직접 탐색하는 대신, 사용자가 입력한 모델명을 표준 DB와 매칭하는 검색 및 선택 인터페이스를 제공하여 데이터의 무결성을 보장한다.

4. 비기능 요구사항 (Non-functional Requirements)

4.1. 성능 및 용량 (Capacity & Scalability)

* 비동기 처리: 외부 API 호출 시 비동기(Asynchronous) 패턴을 적용하여 서버 블로킹을 방지한다.
* 캐싱 전략: Redis 등의 메모리 캐시를 활용하여 빈번한 조회 요청에 대해 100ms 이내의 응답 속도를 유지한다.
* 확장성(Scalability): 트래픽 증가 시 무상태(Stateless) 아키텍처를 기반으로 한 오토 스케일링(Auto-scaling)을 지원하여 수평적 확장이 가능하도록 설계한다.

4.2. 보안성 (Security)

* 인증 표준: 모든 외부 계정 연동 및 로그인은 OAuth 2.0 프로토콜을 준수한다.
* 데이터 암호화: 사용자의 개인정보 및 API 키는 AES-256 알고리즘을 사용하여 암호화 저장한다.

4.3. 신뢰성 및 가용성 (Reliability & Availability)

* 서비스 가동률: 연간 99.9% 이상의 가동률을 목표로 하며, 서비스 장애 발생 시 캐시된 데이터를 우선 노출하는 'Graceful Degradation' 전략을 취한다.
* 우선순위 할당: 프리미엄 구독 사용자의 데이터 동기화 요청에 대해 API 호출 큐(Queue) 우선순위를 할당한다.

4.4. 호환성 및 유지보수성 (Compatibility & Maintainability)

* 브라우저 호환성: HTML5 표준을 준수하여 Chrome, Safari, Edge 브라우저에서의 일관된 동작을 보장한다.
* 유지보수성: Winston 및 Morgan 라이브러리를 통한 상세 로깅 체계를 구축하고, 모듈형 아키텍처를 적용하여 기능 확장을 용이하게 한다.

5. 기타 제약 사항 및 가정 (Constraints & Assumptions)

5.1. 기술 스택 제약

* Frontend: React.js, Tailwind CSS, Recharts
* Backend: Node.js, Express.js
* Database: PostgreSQL (복잡한 하드웨어 관계 모델 및 가중치 쿼리 최적화를 위해 관계형 DB로 확정)

5.2. 환경적 가정

* Steam 및 Riot Games API가 중단 없이 제공되며, 개발자 정책에 따른 데이터 접근 권한이 유효함을 전제로 한다.

6. 수익 창출 및 향후 계획 (Monetization & Roadmap)

6.1. 비즈니스 모델 구현

* 하드웨어 타겟 광고: User_Hardware_Profile 데이터를 분석하여 성능 병목이 예상되는 노후 부품에 대한 업그레이드 제안 및 구매 링크를 노출한다.
* 프리미엄 구독: 인게임 설정 파일(Config)의 클라우드 백업/복원 기능 및 제공자(Power User)와의 심화 성능 비교 데이터를 제공한다.
* 익명화 데이터 판매: 대규모 사용자의 실제 프레임(FPS) 지표를 통계화하여 리서치용 리포트 형태로 B2B 판매를 수행한다.

6.2. 확장 로드맵

* 플랫폼 확장: 향후 에픽게임즈(Epic Games), 콘솔 플랫폼(PSN, Xbox Live) API 연동을 위한 모듈형 커넥터 아키텍처를 추가 개발한다.

7. 부록 (Appendix)

7.1. 용어 사전 (Glossary)

용어	정의
인증된 사용자(Verified User)	OAuth 인증을 통해 하나 이상의 외부 게임 계정 연동을 완료한 사용자.
제공자(Power User)	고성능 하드웨어를 보유하고 신뢰도 높은 최적화 설정값을 등록하는 숙련된 사용자.
수요자(General User)	하드웨어 프로필을 등록하고 타인의 설정값을 조회 및 적용하려는 일반 사용자.
시스템 병목(Bottleneck)	특정 부품의 성능 한계로 인해 전체 시스템의 게임 구동 성능이 제한되는 현상.
Nocuous Ambiguity	구현 단계에서 잘못된 해석을 유발하여 프로젝트에 해를 끼치는 유해한 모호성.

7.2. 수락 기준 (Acceptance Criteria)

ID	요구사항 명칭	수락 기준 (Acceptance Criteria)
AC-01	외부 계정 연동	API 호출 결과 JSON 응답의 모든 인덱스가 DB 필드와 1:1로 매핑되어 저장되어야 함.
AC-02	하드웨어 등록	입력된 모델명이 마스터 DB와 대조되어 정규화된 ID로 변환 저장되어야 함.
AC-03	매칭 알고리즘	가중치(CPU/GPU/Res) 합계가 0.9 이상인 프로필만 필터링되어 정확히 출력되어야 함.
AC-04	데이터 시각화	수신된 JSON 배열의 모든 데이터 포인트가 차트의 좌표와 1:1로 일치하여 렌더링되어야 함.
AC-05	보안성 검증	PII(개인식별정보)가 평문이 아닌 AES-256 암호화 상태로 DB에 저장되어야 함.
AC-06	반응형 레이아웃	최소 해상도(360px)에서 UI 요소의 겹침이나 깨짐 없이 레이아웃이 유동적으로 조정되어야 함.
